The network learned to approximate discrete convolution from raw (rainfall, RTK, time) → RDII data:

RDII ∝ P × R
More rainfall or higher R = more RDII. Double the rain doubles the flow (linearity of convolution)
Peak Lag ≈ Train_peak + T
The RDII peak occurs after the rainfall peak by approximately the UH time-to-peak T
Recession = f(K, Rain shape)
The tail depends on both the UH recession (K) and the rainfall duration — longer rain spreads the RDII response

How RTK MicroGPT v2 Works

This app trains a neural network to learn the convolution of a rainfall hyetograph with the SWMM5 RTK unit hydrograph — the actual computation SWMM5 performs to produce RDII at each node. The network receives 7 inputs (3 rainfall parameters + R, T, K + time) and must predict the convolved RDII flow.

Key difference from v1: Version 1 only predicted the unit hydrograph shape q(t) from R, T, K. That's like teaching someone the shape of a filter without showing them a signal. v2 gives the network real rainfall and asks it to produce the filtered output — discovering the convolution operation on its own through gradient descent.

5

6

Key Concepts Explained
What Is Convolution?

Convolution is the mathematical operation that transforms rainfall into sewer flow. For each time step t, you sum up all past rainfall increments, each weighted by how much of that increment's RDII has arrived by time t. The unit hydrograph (from T and K) tells you how fast each rainfall increment drains into the sewer. This is what the neural network must discover on its own — it never sees the summation loop.

Why 7 Inputs?

3 rainfall inputs describe the storm: P (total depth), D (duration), f (peak position). 3 RTK inputs describe the sewer response: R (how much gets in), T (how fast), K (how long the tail). Plus t (when are we asking?). The network maps this 7D space to a single RDII value.

Triangular Hyetograph

We generate rainfall as a triangle: rising linearly to a peak, then falling. The peak position fraction f controls the shape — f=0.4 means the peak is 40% through the storm (front-loaded, like SCS Type II). The area under the triangle equals P (total depth). This is simpler than real rainfall but captures the essential variation the network must handle.

Why This Is Harder Than Manning's or v1

Manning's: 3 inputs → 1 output, smooth power laws. RTK v1: 4 inputs → 1 output, exponential decay. RTK v2: 7 inputs → 1 output, and the output is a convolution — a summation over time that depends on the entire rainfall history, not just instantaneous parameters. The network must learn both the UH shape AND the aggregation operation.

The Three RDII Triangles in SWMM5

In production SWMM5 models, each node gets three RTK parameter sets: R1/T1/K1 (fast, hours — direct inflow through manhole defects), R2/T2/K2 (medium, hours-to-days — near-surface infiltration), R3/T3/K3 (slow, days-to-weeks — groundwater). Total RDII is the sum of all three convolutions. This app trains on a single triangle — the extension to three is just running three networks in parallel.

How our MicroGPTs relate: Manning's MicroGPT and RTK MicroGPT implement the Value autograd and forward→loss→backward→update loop from microGPT's Utilities and Training Loop boxes. We use a simple MLP instead of Transformer Blocks, but the gradient math is identical. The leap to TinyGPT adds attention, embeddings, and sequence modeling — exactly what a SWMM5 Transformer would need to process full rainfall time series.

Key architectural differences: Our TinyMLP uses ReLU; TinyGPT uses ReLU² (Squared ReLU). We use MSE loss; TinyGPT uses cross-entropy. We predict continuous values (flow); TinyGPT predicts discrete tokens. But the backprop chain rule is the same math everywhere.

The SWMM5 MicroGPT Universe — 16 App Ideas

Every closed-form equation and lookup table in SWMM5 is a candidate for a MicroGPT app. The pattern is always the same: generate perfect training data from the known equation, train a tiny neural network, then show it discovering the physics. Here's the full roadmap, organized by SWMM5 computational module.

Hydraulics & Flow Routing
DONE 1. Manning's MicroGPT
Inputs: n, D, S (3) → Output: Q, V
Q = (1.49/n) × A × R2/3 × S1/2. Full-pipe steady-state flow. The simplest SWMM5 MicroGPT — smooth power laws, 3 inputs, perfect for a first demo.
NEW 2. Partial-Flow MicroGPT
Inputs: n, D, S, d/D (4) → Output: Q, V, A, Rh
Partial depth flow in circular pipes. The Q/Qfull vs d/D relationship has a non-obvious maximum at d/D ≈ 0.938 — can the network discover this peak?
NEW 3. Normal Depth MicroGPT
Inputs: Q, n, S, shape (4) → Output: yn (normal depth)
The inverse Manning's problem — given a flow rate, find the depth. SWMM5 uses iterative bisection. Can a network learn the implicit function directly?
NEW 4. Critical Depth MicroGPT
Inputs: Q, D, shape (3) → Output: yc (critical depth)
When Froude = 1. The specific energy equation E = y + Q²/(2gA²). SWMM5 needs this for mixed-flow transitions and entrance/exit losses.
NEW 5. Weir/Orifice MicroGPT
Inputs: H, Cd, L, type (4) → Output: Q
Transverse, side-flow, and V-notch weirs + orifice equations. The transition from weir to orifice flow is discontinuous — a hard test for ReLU networks.
NEW 6. Pump Curve MicroGPT
Inputs: Hwet well, pump type, curve params (4–6) → Output: Qpump
SWMM5 pump curves (Types 1–4). The network must learn on/off hysteresis, variable-speed relationships, and head-capacity curves.
Hydrology & Infiltration
DONE 7. RTK MicroGPT v2
Inputs: P, D, f, R, T, K, t (7) → Output: RDII(t)
Rainfall × unit hydrograph convolution. The network discovers discrete convolution from scalar inputs. ← You are here
NEW 8. Green-Ampt MicroGPT
Inputs: Ks, ψ, θd, t, i(t) (5) → Output: f(t), F(t)
Infiltration rate and cumulative infiltration. The implicit F equation requires iteration — can the network bypass it? Includes ponding time transition.
NEW 9. Horton MicroGPT
Inputs: f0, f∞, k, t (4) → Output: f(t)
f(t) = f∞ + (f0 − f∞)e-kt. Pure exponential decay — simpler than Green-Ampt but still involves the exponential function the network must approximate with ReLU.
NEW 10. SCS Curve Number MicroGPT
Inputs: P, CN, Ia/S ratio (3) → Output: Qrunoff
Q = (P − Ia)² / (P − Ia + S). SWMM5's simplest runoff method. The threshold behavior (no runoff until P > Ia) creates a non-smooth transition.
NEW 11. SCS Unit Hydrograph MicroGPT
Inputs: Tc, A, Pexcess, t (4) → Output: Q(t)
The dimensionless SCS UH with Tp = 0.6Tc and qp = 484A/Tp. The network learns the standard 22-ordinate dimensionless UH shape and scaling.
NEW 12. Nonlinear Reservoir MicroGPT
Inputs: W, S, n, i(t), ds, t (6) → Output: Qrunoff(t)
SWMM5's subcatchment overland flow. Manning's on a plane: Q = W/n × (d−ds)5/3 × S1/2. Combined with continuity ∂d/∂t = i − e − f − q. The ODE makes this the hardest hydrology MicroGPT.
Water Quality & LID
NEW 13. Buildup/Washoff MicroGPT
Inputs: antecedent dry days, land use, Qrunoff, Cmax, k (5) → Output: pollutant load
Power, exponential, and saturation buildup + EMC/exponential/rating curve washoff. Two coupled processes the network must learn together.
NEW 14. LID Performance MicroGPT
Inputs: P, LID type, berm height, soil Ks, drain coeff (6+) → Output: Qout, retention %
Bio-retention, permeable pavement, rain garden, green roof, infiltration trench. Multi-layer soil + storage + underdrain. The most complex SWMM5 subsystem.
Advanced — Transformer-Scale Ideas
FUTURE 15. SWMM5 Surrogate Model
Inputs: full .inp parameters → Output: peak flows, volumes, timing
Train on thousands of SWMM5 simulation runs. The "holy grail" — a network that replaces SWMM5 for screening-level analysis. Needs TinyGPT-level architecture with attention to handle variable-size networks.
FUTURE 16. RDII Calibration Transformer
Inputs: rainfall time series + observed flow → Output: R1/T1/K1, R2/T2/K2, R3/T3/K3
The inverse problem. A sequence-to-parameters transformer that replaces trial-and-error calibration. This is what RDII Analyst does manually — but learned end-to-end from data across thousands of nodes.

The higher defaults reflect RTK v2's 7-input convolution problem being substantially harder than Manning's 3-input power law. More samples cover the larger input space, more neurons approximate the exponential recession, and more epochs polish the tail.

QWhat is cosine learning rate decay and why does it matter?
▶

The learning rate now follows a cosine schedule from the initial value down to ~0:

LR(epoch) = LRinitial × 0.5 × (1 + cos(π × epoch / max_epochs))

At 500 epochs with LR=0.004, the rate starts at 0.004, is ~0.002 at epoch 250, and approaches zero near the end. The status bar shows the current LR so you can watch it decay.

Why it matters: Without decay, a fixed LR keeps bouncing the loss around instead of settling into minima. The cosine schedule lets early epochs make large steps to learn the overall shape (peak location, rising limb), then late epochs use a tiny LR to polish the exponential recession tail without disturbing the peak fit. This is the same technique used in GPT-4 and all modern LLM training.

QWhat is early stopping and how does it work here?
▶

If the training loss doesn't improve for 80 consecutive epochs, training stops automatically. This prevents wasting time once the network has plateaued.

You'll see two visual cues:

• A ⚠ plateau warning appears in the status bar when stale count exceeds 20 epochs — a heads-up that convergence is slowing

• Training ends with a message showing which epoch it stopped at and the final loss

This means you can safely set Max Epochs to 1000 or 2000 without worrying about wasted compute — the network will stop when it's done learning. With cosine decay, early stopping typically fires between epoch 600–1200 depending on the problem difficulty and hidden layer size.

QWhy 5 samples per event instead of 3?
▶

Each rainfall×RTK event generates a complete RDII hydrograph curve. We then pick random time samples from that curve as training points. With 3 samples, you might get 3 points on the recession and miss the peak entirely. With 5 samples, the odds of covering rise, peak, and recession are much better.

For 6,000 total samples, this means ~1,200 unique rainfall×RTK events with 5 time slices each. Each event has a unique combination of storm depth, duration, shape, and sewer response parameters — giving the network broad coverage of the 7-dimensional input space.

The tradeoff: fewer unique events than 3-per-event (1,200 vs 2,000), but better temporal coverage per event. For convolution learning, temporal coverage matters more because the network needs to understand the full curve shape, not just isolated points.

QWill more epochs help? How many should I use?
▶

Yes, but with diminishing returns. The convolution problem is fundamentally harder than Manning's because the exponential recession tail produces small gradients — the loss drops fast for the peak region but the tail needs many more epochs to refine.

Epochs Expected R² What Improves
250 0.88–0.93 Peak location and rising limb learned; tail still rough
500 0.93–0.97 Recession tail improves noticeably; good balance of speed vs accuracy
1000 0.95–0.98 Tail polish; cosine decay LR is very small, fine-tuning last details
1500+ 0.96–0.98 Diminishing returns — increase hidden neurons to 64–80 instead

With cosine decay, the late epochs use a tiny LR that polishes the tail without disturbing the peak fit. If you're still seeing improvement at 1000 epochs, the bottleneck has likely shifted from training time to network capacity — try bumping hidden neurons to 64–80 before adding more epochs.

Pro tip: Set epochs to 2000 and let early stopping decide. It will automatically stop when the network has converged, typically around 600–1200 epochs with good settings.

The key difference: Manning's equation is an instantaneous function — plug in (n, D, S), get Q. No history, no time. RTK v2's output depends on the entire rainfall history via convolution — a summation over all past time steps. The network must learn both the unit hydrograph shape AND the aggregation operation from scalar inputs alone. That's fundamentally harder for an MLP because convolution is a non-local operation.

QWhat should I try if R² is below 0.90?
▶

Try these fixes in order:

1. More data first: Increase samples to 8000–10000. The 7D input space needs dense coverage.

2. More epochs: Set to 1000+ with early stopping. Let cosine decay do its work on the tail.

3. Bigger network: Try 64–80 hidden neurons. The exponential recession needs more piecewise-linear ReLU segments.

4. Lower initial LR: If loss oscillates wildly, drop to 0.002–0.003. Cosine decay will still anneal it smoothly.

5. Retrain: Random weight initialization matters. Sometimes a fresh run with the same settings gives R² +0.03 just from better initial weights (He initialization is random).

A good target config: 8000 samples, 64 neurons, LR 0.004, 1000 epochs. This should consistently achieve R² > 0.95.

Extensions & Next Steps

Three-triangle RDII: Extend to 12 inputs (P, D, f + R1/T1/K1 + R2/T2/K2 + R3/T3/K3 + t) and train on the combined RDII = sum of three convolutions.

Real rainfall shapes: Replace triangular hyetograph with SCS Type I/IA/II/III distributions, or use recorded rainfall from SWMM rain gauge data.

Inverse calibration: Given observed RDII hydrograph + known rainfall, train a network to predict best-fit R, T, K — the calibration problem RDII Analyst solves.

Sequence model: Replace the MLP with a small LSTM or transformer that takes the full rainfall time series as input instead of summary parameters. This removes the triangular hyetograph simplification.

Multi-event learning: Train on back-to-back rainfall events where antecedent moisture affects the response — capturing the seasonal variation in RDII that makes calibration so challenging.
